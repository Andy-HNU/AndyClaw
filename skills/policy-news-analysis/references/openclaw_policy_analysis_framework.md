# OpenClaw 政策与时事分析框架

> 本文档定义了 OpenClaw 分析任何政府文件或时事内容时应遵循的思维方式、操作工作流，以及背景知识管理机制。
> 目标：不是摘要原文，而是将"执行者语言"翻译成"决策者语言"。

---

## 一、核心认知前提

### 1.1 这是什么

政策与时事分析师指南。服务对象是需要从文件或事件中提取**决策信号**的人。

摘要 ≠ 解读。两者的本质区别是：
- **摘要**：服务于理解，帮助人知道"说了什么"
- **解读**：服务于行动，帮助人判断"意味着什么"、"该怎么做"

### 1.2 通用元框架

无论分析政策文件还是时事内容，底层逻辑链有两种形态，未来我们会有更多的示例，我们可以从中提取并回馈优化我们的逻辑链：

**纵向分解型**（适用于政策文件）
```
异常信号 → 战略定性 → 具体行动 → 政策工具
```

**横向扩散型**（适用于时事分析）
```
异常信号 → 根本原因 → 多方向传导 → 各类资产/行业/主体的影响
```

两种形态的起点相同：**都从找异常开始**。

当分析对象为多份文件（主文件+部委细则）时，优先执行**落差分析**，而不是分别解读后再汇总。

### 1.3 两类分析的共同特征（以实例说明）

| 步骤 | 政策解读示例（政府工作报告） | 时事分析示例（油价暴涨） |
|------|---------------------------|----------------------|
| 找异常 | GDP目标从单点变区间 | IEA释放储备但油价反涨 |
| 解释异常 | 引用官方自注释+历史先例 | 引用海峡过油量数据 |
| 建逻辑链 | 信号→高质量发展→十大任务→宏观政策 | 根因→通胀→美联储→各类资产 |
| 做减法 | 十大任务只深讲三个 | 能源/黄金/科技各取核心一点 |
| 落脚点 | 政策资金流向哪里 | 确定性高的资产在哪里 |

## 二、分析开始前：确认受众问题

**在读任何文件之前，先明确一个前置问题：**
> "这份文件/事件，对【什么类型的决策者】意味着什么？"

| 受众类型 | 核心问题 |
|---------|---------|
| 投资者/资本市场 | 资金会流向哪里？哪些资产被利好/利空？ |
| 产业从业者 | 我所在行业被如何定性？有什么政策窗口？ |
| 企业经营者 | 营商环境有何新变化？有哪些合规要求改变？ |
| 研究者/媒体 | 今年的政策重心与去年相比有何变化？ |

**操作规则**：每次分析前，先向用户确认受众类型。受众类型决定你过滤哪些内容、保留哪些信息。

## 三、第零步：背景知识获取（读文件之前执行）

> 分析质量的上限由背景知识决定。在正式读文件之前，先诊断知识缺口，定向补充。

### 3.1 知识需求诊断

拿到分析任务后，快速扫描标题和前200字，预判需要哪类背景知识：

**三类知识，获取成本不同**

| 类型 | 特征 | 获取方式 |
|------|------|---------|
| ① 结构性背景知识 | 变化慢，长期有效 | 查询 SQLite `domain_knowledge` 表 |
| ② 当前状态知识 | 定期更新 | 查询 SQLite `current_state` 表 |
| ③ 事件触发知识 | 分析特定文件时才需要 | 分析时动态补充/标记缺口 |

### 3.2 SQLite 知识库结构

```sql
-- 表1：领域基础知识（结构性，长期有效）
CREATE TABLE domain_knowledge (
  id          INTEGER PRIMARY KEY,
  domain      TEXT,        -- 'energy' / 'macro' / 'policy' / 'industry'
  topic       TEXT,        -- 'strait_of_hormuz' / 'fed_rate_mechanism'
  content     TEXT,        -- 具体知识内容
  key_numbers TEXT,        -- 关键数字，JSON格式
  updated_at  DATE
);

-- 表2：历史参照库（用于"这次vs历史"对比）
CREATE TABLE historical_reference (
  id             INTEGER PRIMARY KEY,
  event_type     TEXT,    -- 'oil_crisis' / 'gdp_target' / 'rate_hike'
  year           INTEGER,
  description    TEXT,
  outcome        TEXT,    -- 当时的结果
  analogy_power  INTEGER  -- 1-5，类比价值评分
);

-- 表3：当前状态快照（定期更新）
CREATE TABLE current_state (
  id          INTEGER PRIMARY KEY,
  category    TEXT,    -- 'macro' / 'market' / 'policy'
  key         TEXT,    -- 'fed_rate' / 'oil_price_range' / 'china_gdp_target'
  value       TEXT,
  context     TEXT,    -- 简短说明
  valid_until DATE     -- 这条知识的有效期
);
```

### 3.3 定向查询流程

```
步骤1：识别领域标签（可多选）
  [macro宏观] [energy能源] [policy政策] [industry产业]
  [market市场] [geopolitics地缘] [tech科技]

步骤2：列出"知识缺口假设"（每类最多3条）
  - 可能缺：[知识类型] → 查 domain_knowledge WHERE domain=X AND topic=Y
  - 可能缺：[历史参照] → 查 historical_reference WHERE event_type=X
  - 可能缺：[当前状态] → 查 current_state WHERE category=X AND key=Y

步骤3：执行定向查询，不做全表扫描
  把查询结果作为"背景知识包"，带入后续分析步骤
```

### 3.4 知识缺口处理规则

| 情况 | 处理方式 |
|------|---------|
| 查到了 | 直接使用，在输出中标注 `[数据库]` |
| 没查到，但判断重要 | 在输出末尾标注 `[知识缺口]`，说明缺什么 |
| 没查到，不确定重要性 | 用 `[待验证]` 标记相关判断，继续分析 |
| **绝对禁止** | **不能虚构数字，不能用模糊表达掩盖知识缺口** |

## 四、七步分析工作流 + 第八步交付合成

> 前七步是分析引擎日志；第八步是面向决策者的交付摘要。
> 禁止仅输出前七步日志而缺少第八步。

### 第一步：异常值扫描

扫描整份文件/事件，寻找"偏离常规"的表达或现象。

**异常值的三种形态**

1. **措辞异常**：新出现的词汇、术语，或消失的往年惯用表达
2. **结构异常**：排序变化（去年第三今年第一）、新增或删除的章节
3. **数字/现象异常**：区间代替单点、数字明显偏高/偏低、反常的市场反应

**操作方法**
- 如有往年同类文件，对比阅读，标记差异
- 如无往年文件，关注带有副词的句子：
  - "首次"/"新" = **增量信号**，重点解读
  - "继续"/"深化" = **延续信号**，看宾语是否有新变化
  - "反而"/"不降反涨" = **市场异常信号**，挖根因

**输出格式**：列出3-5个异常点，每个写明"异常内容"和"为什么算异常"。

---

### 第二步：让文件/事件自己解释自己

> **前置确认**：执行本步骤前，确认素材类型（A类/B类/C类，见卷一定义）。
> B类解读稿的"官方注释"是记者整理或受访官员口头表述，不等于正式条款，
> 引用时须标注 `[B类解读稿，非原文条款]`，不得与 A 类原文混用。

**政策文件**：通常在提出目标后紧跟官方自注释，寻找"考虑是……"、"主要原因是……"等句式。这些是最高密度的一手信息。

**时事内容**：寻找事件内部的因果线索——哪个数据解释了哪个现象，哪个行为说明了哪个判断。

**你的任务是**：
1. 找到这些官方注释或内部因果线索
2. 用更通俗的因果语言复述
3. 补充官方注释中省略的背景数字或历史数据（调用知识库）


### 第三步：从外部资料中解释异常

第二步是让文件自己解释自己，但文件的自注释往往只说"是什么"，不说"为什么严重"。这一步的任务是调用外部资料，给异常定量、定性、定历史坐标。

**触发条件**：当第二步的官方注释或事件内部线索无法充分解释异常的严重程度或背景时，必须执行此步骤。

**外部资料的三个用途**

| 用途 | 操作 | 示例 |
|------|------|------|
| **定量**：给异常配上数字 | 查知识库或标注缺口 | "区间目标历史仅用过两次"、"霍尔木斯每日过油2000万桶→封堵后不足30万桶" |
| **定性**：判断异常的严重程度 | 与历史均值对比 | "4亿桶储备够全球用4天，历史最大单次释放是1.8亿桶，依然杯水车薪" |
| **定坐标**：找历史先例锚定 | 查 `historical_reference` 表 | "上次用区间目标是2019年，背景是中美贸易摩擦+内部结构调整" |
**操作步骤**
1. 拿出第一步识别的异常点列表

2. 对每一个异常点，判断：
   a. 文件/事件本身已经充分解释了吗？（第二步已覆盖）
      → 是：跳过
      → 否：进入外部查询

3. 执行定向查询（优先查 SQLite，其次标注缺口）
   - 查 historical_reference：这个异常类型历史上出现过吗？上次的背景和结果是什么？
   - 查 domain_knowledge：有没有能给这个异常定量的行业/领域基础数据？
   - 查 current_state：当前的宏观/市场背景是否放大了这个异常的影响？

4. 用外部资料"包裹"异常
   输出格式：
   - 异常现象：[第一步发现的异常]
   - 内部解释：[第二步找到的官方注释]（如有）
   - 外部定量：[数字支撑，来源标注 数据库 或 待验证]
   - 历史坐标：[最近一次类似异常是何时、背景是什么]
 （如历史坐标来自推断而非知识库命中，标注 [推断] 而非 [数据库]，两者不可混用）
   - 严重程度判断：[低/中/高]，一句话说明理由
**与第七步的区别**

第三步是**针对异常的精准查询**，目的是"解释为什么这个信号值得重视"；第七步是**全局性的知识校准**，目的是"丰富整个逻辑链的可信度"。两者都用外部知识，但触发时机和覆盖范围不同。

---

### 第四步：从排序和结构中读优先级

政策文件的排列顺序不是随机的，是权重的隐性表达。

**操作步骤**
1. 找出文件中所有"并列枚举"的段落（如"十大任务"、"四个突出"）
2. 记录每项的顺序位置
3. 与往年对比：上升了几位？下降了几位？新增了哪项？
4. 重点深挖排在**第一位**的项目

**注意**：不要平等对待所有并列项目。排第一位的优先解读，中间项目如与第一位有逻辑关联则保留，末位项目可以简化处理。

---

### 第五步：用逻辑完整性做减法

文件涵盖领域极广，必须做减法。减法的标准不是"重不重要"，而是**是否构成逻辑闭环**。

**操作步骤**
1. 从受众核心问题出发，列出"必须回答的子问题"
   - 例（对投资者）：供给侧怎么了？需求侧怎么了？钱从哪来？
   - 例（对产业从业者）：我的行业被定性为什么？有哪些支持政策？有哪些风险？
2. 从文件中找到回答这些子问题的段落
3. 检验这些段落是否能构成自洽的逻辑链
4. 逻辑链已经闭合后，剩余内容全部压缩处理

**输出格式**
```
核心逻辑链：[A] → [B] → [C]
核心段落：第X节、第Y节、第Z节
压缩处理的段落：（章节名列表）
```

减法边界说明（防止收口过紧）

减法的目的是去掉与受众核心问题无关的内容，不是压缩所有细节。
以下情况不应被减掉：

- 同一根因向多个产业传导的横向影响
 （参考横向扩散型分析：油价涨→能源/黄金/科技分别受影响，三条线都要保留）
- 政策工具涉及多个执行抓手，每个抓手指向不同赛道
 （如财政工具分消费/设备更新/银行资本补充，三个方向影响不同标的，不能合并）
- 文件中明确列出的并列领域
 （原文单独列出的领域，减法不能把它们合并或省略）

做完减法后执行一次自检：
逻辑链上是否有分叉节点被切掉了？
如果有，恢复那个分叉，作为并行支线保留在输出里。

---

### 第六步：识别"变"与"不变"

这是解读中最容易被忽略、也最有信息量的步骤。**执行深度要求下探到措辞层，不能停在定调层。**

**寻找三类变化**

| 变化类型 | 示例 | 含义 |
|---------|------|------|
| 定调变化 | 去年"稳健"→今年"积极" | 方向性改变，最高优先级 |
| 措辞变化 | 加了"定向"、"结构性"、"差异化" | 政策从普惠转向精准，资金流向更集中 |
| 数字变化 | 区间代替单点 | 策略上的主动留白 |

**易犯错误**：找到"定调不变"就停手，忽略同一定调下的措辞微变。例如货币政策定调"适度宽松"与去年相同，但"优化创新结构性货币政策工具"是新增表述——这是真正的新信号，不能归入"不变"。

**操作方法**：把今年的关键政策表述与去年逐句对比，画出变化矩阵：

```
         | 变化 | 不变 | 新增
---------|------|------|----
定调     |      |      |
措辞     |      |      |
数字     |      |      |
```

---

### 第七步：用外部知识校准

原文提供"是什么"，外部知识提供"为什么重要"。

### 第八步：合成投资决策摘要（交付层）

将前七步日志压缩为“投资者可直接使用”的一页摘要。
该步骤必须回答三件事：
1. 这件事对投资者意味着什么（本质）
2. 受益/受损/暂不判断方向分别是什么（含确定性）
3. 现在该怎么做、等什么再做（行动与观测点）

禁止让用户自行从七步日志拼结论。

**需要调用的外部知识类型**
1. **历史先例**：这个做法上次什么时候用过？当时是什么背景？结果如何？
2. **基准数据**：文件中的数字相对历史均值是高是低？
3. **外部环境**：国际经贸、地缘政治对某些表述的解释力
4. **行业知识**：特定领域的政策表述对那个行业意味着什么

**重要规则**：外部知识用于"定性"，不能用于"篡改"原文意思。如果不确定某个外部数据，明确说明"需要验证"，不要虚构。

## 五、最终输出结构

### 输出排版规范（每次输出必须遵守）

所有分析输出按以下固定顺序和分隔格式呈现，不得调换顺序：

---

📋 投资决策摘要｜交付内容
─────────────────────────

[投资决策摘要内容]

─────────────────────────
⚙️ 分析日志（Step 0–7）｜供存档备查，非交付内容
─────────────────────────

[七步日志内容]

─────────────────────────
💾 知识沉淀包
─────────────────────────

[知识沉淀包内容 + 写库回执]

---

**排版规则说明**：

- 三个区块必须全部输出，缺一不可
- 📋摘要放第一位，投资者拿到输出第一眼即看到结论，无需翻找
- ⚙️日志区块是质量保障机制，不是交付物，但必须保留供审计
- 💾知识沉淀包始终放最后，不干扰阅读流


### 标注规范（贯穿所有输出）

输出中必须区分四类信息的来源，混用是质量事故：

| 标注 | 含义 | 使用场景 |
|------|------|---------|
| `[数据库]` | 来自知识库的背景支撑 | 引用 SQLite 查询结果时 |
| `[推断]` | 分析师自行延伸的判断，原文无直接依据 | 从信号推导行动建议时 |
| `[待验证]` | 判断依赖外部数据，尚未确认 | 数据来源不确定时 |
| `[知识缺口]` | 缺少关键背景知识，影响判断质量 | 查库未命中且判断重要时 |

**凡是原文没有直接对应的推断，必须标注 `[推断]`，不得混入原文结论。**

```markdown
## 核心信号摘要（3句话以内）
本次文件/事件传递的最重要的1-2个新信号

## 信号来源（第1-3步的发现）
- 异常点是什么
- 排序/结构变化说明了什么

## 核心逻辑链
[信号] → [战略定性/根因] → [具体行动/传导路径] → [政策工具/资产影响]

## 值得深挖的2-3个关键点
每个写明：
- 原文/事件表述
- 真正含义
- 对受众的行动意义

## 延续未变的部分（简要列出）
哪些是常规延续，不需要过度解读

## 知识缺口与待验证事项
哪些判断依赖外部数据、需要后续事件印证
标注 [知识缺口] 或 [待验证]
```

### 受众专项输出：投资决策摘要（投资者视角专用）

第八步完成后，必须额外生成本节摘要。
前七步日志是分析过程记录，留存备查；本摘要是交付给投资者的实际内容。
两者都要输出，不能用日志代替摘要，也不能只输出摘要跳过日志。

---

## 投资决策摘要｜[文件名]｜[日期]

这件事的本质
> 先用1句话说清楚政策/事件的性质（方向性表态、执行工具升级、还是增量政策）。
> 如果信号辐射多个产业，逐个列出，不强行合并成一句：
>
> - [产业A]：[这个信号对它意味着什么，1-2句]
> - [产业B]：[这个信号对它意味着什么，1-2句]
> - [产业C]：[这个信号对它意味着什么，1-2句]
>
> 判断标准：信号与该产业有直接因果关系才列入；
> 间接关联或需要多步推导的放到下方"市场影响判断"表格，不在此处展开。

市场影响判断

| 维度 | 判断 | 来源 | 确定性 |
|------|------|------|--------|
| 受益方向 | [具体行业/企业类型] | [原文/推断] | 高/中/低 |
| 受损方向 | [具体行业/企业类型] | [原文/推断] | 高/中/低 |
| 暂无方向 | [说明原因] | — | — |

现在该怎么做

- 可以做：确定性高、有原文支撑的行动方向
- 等待做：需要等后续文件或数据验证后再判断（注明等什么）
- 不用做：本次信号与哪些方向无关，避免过度解读

最重要的后续观测点
> 写1-2个最关键的验证节点。
> 格式：等[什么机构]发布[什么文件]，重点看[什么指标或条款]。

标注规则：
- 无标注 = 有原文直接支撑
- [推断] = 从信号延伸的分析师判断
- [待验证] = 需后续文件或数据确认

## 六、知识库持续沉淀机制

> 每次分析完成后，同步输出"知识沉淀包"，写回 SQLite。
> 目的：分析的第1篇和第100篇消耗相同token，但第100篇质量远高于第1篇。

**强制规则：知识沉淀包必须展示每条记录的完整内容，禁止只报告入库计数。**
只报"domain_knowledge: 3条"而不列出内容，等于允许错误数据静默写入，会污染所有后续分析。

**每次分析结束后，输出以下内容**：

> 额外强制：知识沉淀 JSON 每条记录都必须带来源注释字段，至少包含：
> `source_type`（A政策原文/B官方解读/C二手报道）、`source_url`、`source_title`、`source_quote`（对应原文摘句）、`source_date`（可得时）。
> B类引用必须附注：`[解读稿引用，非原文条款]`；C类不得入库。

```
## 知识沉淀包（本次分析产出）

### 新增 domain_knowledge（本次用到的背景知识）
- domain: [领域]
- topic: [主题]
- content: [内容]
- key_numbers: [关键数字，JSON]
- source_type: [A类/B类] ← 必填，对应卷一素材类型定义

### 新增 historical_reference（本次引用的历史参照）
- event_type: [类型]
- year: [年份]
- description: [描述]
- outcome: [结果]
- analogy_power: [1-5]

### 更新 current_state（本次涉及的当前状态数据）
- category: [类别]
- key: [键名]
- value: [值]
- context: [说明]
- valid_until: [有效期]
```

**写库回执格式规范**：必须区分本次新增与库存累计，禁止只报累计总数。

标准格式：
{
 "本次新增": {"domain_knowledge": X, "historical_reference": X, "current_state": X},
 "库存累计": {"domain_knowledge": X, "historical_reference": X, "current_state": X}
}

## 七、知识缺口主动填充工作流

> 当前框架教会了识别缺口，但没有定义发现缺口之后做什么。
> 本节定义从"打标签"到"主动闭环"的完整流程。

### 7.1 触发条件：缺口分级

不是所有缺口都值得触发查询。先判断缺口的位置：

| 缺口位置 | 处理方式 |
|---------|---------|
| 卡在逻辑链的**关键节点**（无此知识则核心判断无法成立） | 必须触发主动填充（硬约束，不可仅打标签后跳过） |
| 影响**细节丰富度**（有此知识更好，但不影响核心结论） | 标注 `[待验证]`，继续分析，不触发查询 |
| **边缘信息**（与受众核心问题无关） | 忽略，不标注 |

### 7.2 查询工具优先级

openclaw 有两级查询能力，按顺序使用，避免跳级：

```
第一级：SQLite 知识库（零额外成本）
  → 查 domain_knowledge / historical_reference / current_state
  → 命中：直接使用，标注 [数据库]
  → 未命中：进入第二级

第二级A：内置搜索（快，适合明确的事实性查询）
  → 适用场景：数字核实、机构官方立场、近期事件
  → 单次查询，不做反复迭代
  → 命中：评估质量后使用，标注 [外部补充]
  → 结果不足：进入第二级B

第二级B：agent-browser（深，适合需要完整页面内容的查询）
  skill: https://clawhub.ai/MaTriXy/agent-browser-clawdbot
  → 适用场景：官方全文、报告原文、需要导航多页的数据
  → 操作示例：
      agent-browser open <url>
      agent-browser snapshot -i      # 获取可交互元素
      agent-browser click @e1        # 点击目标元素
  → 命中：提取关键内容，标注 [外部补充:browser]
  → 仍未命中：降级为 [知识缺口]，不强行填充
```

### 7.3 查询构造规范

把缺口描述转化为可执行的查询意图，禁止用模糊词搜索：

| 缺口描述（模糊，不可搜） | 查询意图（精准，可搜） |
|----------------------|-------------------|
| "霍尔木斯海峡相关情况" | "strait of hormuz daily oil flow barrels 2024" |
| "历史上GDP区间目标" | "中国 GDP 目标 区间 2016 2019 背景" |
| "美联储当前利率" | "fed funds rate current 2026" |
| "IEA历史最大储备释放" | "IEA strategic petroleum reserve largest release history" |

**构造原则**：缺口名词 + 关键数字维度 + 时间限定词。

### 7.4 结果质量评估

查到资料不等于可以直接用，先过三关：

```
关卡1：来源可信度
  权威来源（IEA / 央行 / 国家统计局 / 政府官网）→ 直接使用
  主流媒体报道 → 使用但标注 [待验证]
  论坛/自媒体/无出处数据 → 不使用，降级为 [知识缺口]

关卡2：时效性
  数据是否在有效期内？（current_state 表有 valid_until 字段可参考）
  过期数据用于"历史参照"可以，用于"当前状态判断"不可以

关卡3：与逻辑链的一致性
  新数据与已建立的逻辑链是否冲突？
  → 不冲突：写入并使用
  → 冲突：先标注冲突，呈现两种可能，不强行选一个
```

### 7.5 填充后的动作

```
质量通过 →
  ① 用于当前分析，在相关判断处标注 [外部补充] 或 [外部补充:browser]
  ② 写入 SQLite 对应表（domain_knowledge 或 current_state）
  ③ 在知识沉淀包中列出完整记录

质量存疑 →
  ① 用于当前分析，标注 [待验证]
  ② 不写入 SQLite（避免污染）
  ③ 在知识缺口节说明需要后续人工核实

查询失败 →
  ① 保留 [知识缺口] 标注
  ② 在输出末尾说明：缺口内容 + 建议人工补充的来源
  ③ 明确说明此缺口对核心结论的影响程度（高/中/低）
```

### 7.6 禁止行为

- **禁止用模糊表达掩盖缺口**：不能写"相关数据显示……"而不给出来源
- **禁止因查询失败而虚构数字**：宁可标注缺口，也不能编造数据
- **禁止跳过质量评估直接写库**：未经评估的数据写入会污染后续所有分析
- **禁止对边缘缺口触发 agent-browser**：agent-browser 成本最高，只用于关键节点的深度查询


## 九、整体运行架构

```
用户给文件/事件
        ↓
确认受众类型和核心问题
        ↓
第零步：知识需求诊断
  → 识别领域标签
  → 列出知识缺口假设
  → 定向查询 SQLite（domain_knowledge / historical_reference / current_state）
  → 获得"背景知识包"
        ↓
七步分析工作流
  第一步：异常值扫描
  第二步：让文件自己解释自己
  第三步：从外部资料中解释异常
           ↓ 遇到关键节点知识缺口
           知识缺口主动填充工作流（第七节）
             第一级：查 SQLite
             第二级A：内置搜索
             第二级B：agent-browser（深度查询）
             质量评估 → 写库 or 标注[待验证] or 保留[知识缺口]
           ↑ 填充完成，返回分析流程
  第四步：从排序读优先级
  第五步：用逻辑完整性做减法
  第六步：识别变与不变（下探到措辞层）
  第七步：用外部知识校准
        ↓
输出解读报告
  标注规范：[数据库] / [推断] / [待验证] / [知识缺口] / [外部补充] / [外部补充:browser]
        +
输出知识沉淀包（完整内容，非计数）→ 写回 SQLite
```

### 案例A：纵向分解型（2026年政府工作报告）

```
异常信号：GDP目标首次采用区间（4.5%-5%），历史上仅2016、2019年用过
    ↓
战略定性：高质量发展是主轴，排十五五四个突出之首
    ↓
具体行动：
  - 任务二（新动能）→ 四个抓手：传统产业升级/新兴产业/AI经济/服务业
  - 任务三（科技自立）→ 关键核心技术攻关，技术自主可控
  - 首要任务（内需）→ 投资+消费双轮驱动
    ↓
政策工具：财政定向投放，货币结构性支持（方向>量级）
```

**关键提取技巧**：
- 区间目标异常 → 用历史先例定性（2016/2019背景）
- 官方自注释"为调结构、防风险留空间" → 翻译为具体的数字逻辑（下限高于4.1%最低线）
- "继续实施"+"结构性" → 定调不变但方向精准化是新信号

### 案例B：横向扩散型（油价暴涨分析）

```
异常信号：IEA释放4亿桶储备，历史上释放必降油价，但这次反涨
    ↓
根本原因：市场认为美国已失去对霍尔木斯海峡局势的掌控
    ↓
多方向传导：
  ├── 能源板块：石油贵→传统能源贵→新能源替代效应激活
  │            铝：中东电解铝占9%全球产能，海峡封堵导致供给收缩
  │            海上风电：欧洲能源安全需求加速新能源投资
  ├── 黄金：油价涨→通胀预期升→降息预期降→美联储不降息→流动性收紧→黄金横盘
  └── 科技股：通胀预期升→利率预期升→估值压制→科技股跌
    ↓
行动建议：不确定性下选"黑铁资产"——同时拥有高成长性和高防御性
  例：电力基础设施（AI需求→电力需求，确定性高，无泡沫风险）
```

**关键提取技巧**：
- 找市场异常反应作为起点（涨而不跌）
- 用具体数字建立因果（每天2000万桶→不足30万桶）
- 横向扫描所有受影响资产，找到共同根因（霍尔木斯失控）
- 落脚到"确定性"而非"最高收益"

### 案例C：多文件对照分析（政策主文件 vs 落地细则）

**适用场景**：主文件（政府工作报告/规划纲要）发布后，配套部委细则陆续出台，需要判断政策是否真正落地。

核心问题不是“细则说了什么”，而是三个落差：

1. **落差1：有承诺无配套**
   报告说要做 → 细则里没有对应条款  
   → 信号：该方向优先级可能低于表述，或执行在观望

2. **落差2：有配套有超预期**
   报告未提及 → 细则里悄悄出现新内容  
   → 信号：这才是真正的增量信息，往往比报告本身更值得关注

3. **落差3：有配套但力度缩水**
   报告措辞强 → 细则条款模糊/设门槛/限定范围  
   → 信号：政策意图和执行能力之间存在约束，需要降低预期

**执行步骤**

**Step 1：建立“承诺清单”**
- 从主文件中提取所有明确政策承诺，列成清单
- 格式：`[机构] + [动作] + [对象] + [时间]`
- 例：`财政部 → 发行 → 超长期特别国债1.3万亿 → 2026年`

**Step 2：逐条比对细则**
- 对每条承诺，在部委细则中查找对应条款
- 标注：`✅ 有配套 / ❌ 无配套 / ⚠️ 力度缩水`

**Step 3：扫描细则增量**
- 通读细则，标出所有主文件中未出现的新内容
- 标注：`[细则新增]`

**Step 4：输出落差报告**
- 重点呈现：`❌ 无配套项 + [细则新增]项`
- 这两类信息密度最高

**输出格式示例**

```markdown
## 落差分析：[主文件名] vs [细则名]

### ❌ 有承诺无配套（共X项）
- [承诺内容]：细则中未见对应条款 → [推断]可能原因

### ✅ 对照一致（共X项，简列）
- [承诺] → [细则条款简述]

### ⚠️ 力度缩水（共X项）
- [报告措辞] vs [细则实际表述] → 差距说明

### 🆕 细则新增（共X项，重点关注）
- [新内容]：主文件未提及 → [推断]信号含义
```

*本框架适用于：政府工作报告、央行政策声明、行业规划文件、地缘政治事件、大宗商品市场异常等任何需要从信息中提取决策信号的分析场景。*
